# 0006. スケジューラーサービス実装

日付: 2025-06-22

## ステータス

採用

## 文脈

ぱすぱるアプリでは、端末がスリープ中でも課題差分取得や期限前リマインドなどのバックグラウンドタスクを確実に実行する必要がある。Android と iOS では異なるバックグラウンド処理APIを使用するため、プラットフォーム固有の実装を抽象化する統一インターフェースが必要である。

## 検討した選択肢

1. **各機能で直接プラットフォームAPIを呼び出す**
   - 利点: シンプルで直接的
   - 欠点: コードの重複、プラットフォーム固有のロジックが散在、テストが困難

2. **単一の統一サービスクラス**
   - 利点: 1つのクラスで全てを管理
   - 欠点: プラットフォーム固有のロジックが混在、保守性が低い

3. **抽象インターフェースとプラットフォーム固有実装の分離**
   - 利点: クリーンな設計、テスタビリティ、保守性
   - 欠点: わずかに複雑

## 決定

抽象インターフェース `SchedulerService` とプラットフォーム固有実装（`AndroidScheduler`、`IosScheduler`）を分離したアーキテクチャを採用する。

### 実装詳細

#### 共通インターフェース

- `SchedulerService` 抽象クラス
- `scheduleOneOff`: 一回限りのタスクスケジュール
- `schedulePeriodic`: 定期実行タスクスケジュール  
- `cancel`: タスクキャンセル

#### Android実装 (`AndroidScheduler`)

- WorkManager プラグインを使用
- `OneOffWorkRequest` / `PeriodicWorkRequest` でタスクスケジュール
- バッテリー最適化を考慮した制約設定
  - `requiresBatteryNotLow: true`
  - `networkType: NetworkType.connected`

#### iOS実装 (`IosScheduler`)

- Background Fetch プラグインを使用
- `BGAppRefreshTaskRequest` でバックグラウンド更新要求
- 定期タスクは一回限りタスクの再スケジュールで実現

#### 依存性注入

- Riverpod `Provider` でプラットフォーム検出に基づく適切な実装を提供
- `Platform.isAndroid` / `Platform.isIOS` による自動選択

#### 初期化

- Android: `main()` で `Workmanager().initialize()` 実行
- iOS: `main()` で `IosScheduler.initialize()` 実行
- コールバック関数での タスク実行ロジック分岐

## 影響

### 正の影響

- **保守性**: プラットフォーム固有ロジックの分離
- **テスタビリティ**: モックによる単体テスト可能
- **拡張性**: 新しいプラットフォーム対応が容易
- **可読性**: 責任の明確な分離

### 負の影響

- **複雑性**: 単純な実装に比べてファイル数増加
- **学習コスト**: 新しい開発者がアーキテクチャを理解する必要

### 影響を受ける機能

- 課題差分取得バックグラウンドタスク
- リマインダー通知スケジュール
- データ同期タスク
- その他の定期実行機能

## 注記

- iOS のバックグラウンド処理には厳しい制限があるため、実際の動作は端末設定やアプリ使用頻度に依存する
- Android の WorkManager は端末のDoze モードやバッテリー最適化設定の影響を受ける
- 本実装はベストエフォート型であり、100%の実行保証はできない
